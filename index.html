Token Philosophy and Logic

How and why this system is structured.

Core Philosophy

Themes should change personality without rewriting components. Components should never directly depend on fixed colors or one-off dimensions. 

Component class
  -> component tokens (button/input/card)
     -> semantic tokens (bg/text/border/accent)
        -> theme values

Token Creation Rules

Create tokens when the value has reuse potential or must vary by theme family. 

Use token when:
- value appears in 2+ components
- value needs theme variation
- value represents intent (warning, accent)

Avoid token when:
- one-off layout local to one element

Preset Architecture

Builder presets are organized by categories so combinations are composable. 

Color Family + Hue -> full semantic palette behavior
Scheme -> component-level color logic and emphasis behavior
Style -> geometry, borders, shadows, and interaction language
Typography -> font voice, type scale, and reading rhythm
Scale -> spacing + control density
Texture -> optional dynamic material layer

Final custom theme = Family + Hue + Scheme + Style + Typography + Scale + Texture + manual edits

Preset Effect Map

How each preset layer modifies tokens and aliases. 

Color Family:
- contrast model (light/dark, mono/bold)
- semantic intent tuning (success/warning/danger)

Hue:
- sets primary hue anchor
- combines with family behavior to generate semantic palette

Style:
- shape, border language, shadows, interaction depth

Scheme:
- component color behavior (button/input/chip treatments)
- expressive palette logic for poster, terminal, EVA, alloy, etc.

Typography:
- font family pairings
- typographic scale and line-height behavior

Scale:
- density, control height, spacing cadence only

Texture:
- optional dynamic layer (grain, metal, glass, wire)
- drives image overlays, blend, and backdrop blur

Alias flow:
component token -> semantic token -> foundation token
ex: --ds-btn-bg -> --ds-bg-raised -> --ds-bg

Preset combo examples:
- paper + green + mint-ink + paper + neutral-ui + standard + clean (paper mint)
- cathode + amber + amber-glow + terminal-amber + terminal-glow + compact + amber-screen (amber terminal)
- signal-paper + orange + signal-poster + graphic-signal + poster-grotesk + standard + poster-paper (signal poster)
- signal-paper + orange + signal-diagram + graphic-diagram + poster-grotesk + standard + poster-paper (signal diagram)
- signal + violet + eva-unit01 + eva-unit01 + eva-unit-sans + compact + eva-unit-base (eva unit-01)

Modularization note:
- amber terminal keeps its CRT identity through scheme/style/typography/texture while alias tokens derive from semantic tokens, so hue swaps can recolor it cleanly.

Fidelity rule:
- if changing one preset category alters unrelated properties, fix the preset layer that owns the drifting token.
- ex: scale should change density and sizing tokens, not palette tokens.

Button Contrast Logic Matrix

Source-of-truth logic for primary/selected/readability issues. 

Case                         | Token/Rule Source                               | Expected Text Logic
Primary default              | --ds-btn-primary-text                           | Theme text strategy (semantic)
Primary hover (global)       | --ds-btn-primary-bg-hover + primary text token  | Keep readable against hover fill
Primary selected             | .ds-btn-primary[aria-pressed="true"]            | Use primary token stack (not generic selected)
Secondary selected           | .ds-btn[aria-pressed] (non-primary variants)     | Accent-leaning selected treatment
Checkbox/radio checked mark  | --ds-check-mark                                  | Contrast against --ds-check-checked-bg
Tab selected label           | --ds-tab-active-fg / --ds-tab-active-bg          | Contrast independent of primary button text

Theme-specific exception policy:
- Bright-primary themes should use a negative-space label strategy: map primary text and checked marks to a dark surface token such as --ds-bg-elevated.
- If a theme's highlight states lose contrast, patch at theme token level first.
- Add component-level theme-scoped override only when semantic tokens are not enough.

Implementation Pattern

For new components, define local component tokens first, then map them to semantic tokens. 

.ds-new {
  background: var(--ds-new-bg);
  border-color: var(--ds-new-border);
}

:root {
  --ds-new-bg: var(--ds-bg-raised);
  --ds-new-border: var(--ds-border);
}

Native Controls and Overlay Behavior

Native Controls and Viewport Scrollbar

Number inputs, checkboxes, radios, range sliders,
and color pickers are tokenized via:
- --ds-color-input-*
- --ds-check-*
- --ds-range-*
- --ds-spin-*

Page scrollbar theming:
- html/body scrollbar uses --ds-scrollbar-*
- local scroll regions use .ds-scroll + same tokens

Note:
Browser chrome (tab bar / app frame scrollbar) is OS/browser UI
and cannot be themed from site CSS.

Modal and Popover Placement

Modal behavior:
- .ds-modal uses full-viewport centering
- .ds-modal-panel is centered with no fixed top offset

Popover behavior:
- Auto places below trigger by default
- Flips above when viewport space below is insufficient
- Horizontal alignment shifts start/end to avoid clipping

Goal:
Do not occlude the trigger element when contextual help opens.

Using This Repository In Projects

GitHub Workflow

1. Clone this repository
2. Copy _DesignSystem into your app repo
3. Link your app HTML:
   <link rel="stylesheet" href="./_DesignSystem/theme.css">
4. Add theme package scripts:
   <script src="./_DesignSystem/js/theme-registry.js"></script>
   <script src="./_DesignSystem/js/theme-selector.js"></script>
5. Add a selector:
   <select data-ds-theme-select data-ds-theme-storage="app-theme"></select>
6. Set the theme on root:
   <html data-theme="steel-night">
7. Append custom exported themes to:
   _DesignSystem/css/themes.css
8. Add new theme metadata to:
   _DesignSystem/js/theme-registry.js

Local Package Workflow

1. Build your theme in Theme Builder
2. Click Generate Export
3. Click Download Theme Package
4. Drag extracted files into your project folder
5. Follow INSTALL.md in the package

App Integration Snippet

<html lang="en" data-theme="steel-night">
<head>
  <link rel="stylesheet" href="./_DesignSystem/theme.css">
</head>
<body>
  <label for="themeSelectApp">Theme</label>
  <select
    id="themeSelectApp"
    data-ds-theme-select
    data-ds-theme-storage="projektik-theme"
  ></select>

  <script src="./_DesignSystem/js/theme-registry.js"></script>
  <script src="./_DesignSystem/js/theme-selector.js"></script>
</body>
</html>
